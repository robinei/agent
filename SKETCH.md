# Agent Server — Design Document

## Overview

A personal coding agent server running on a home PC, accessible from anywhere via
Tailscale. A browser-based frontend (PWA) provides the UI. The agent has full access
to the local filesystem, shell, git, and Python. All intelligence comes from the
Anthropic API.

The goal is to continue working on hobby repos from an iPhone when away from home,
directing the agent by voice and watching it work.

---

## Architecture

```
iPhone (Safari PWA)
  └── Tailscale → Home PC

Home PC
  ├── Python agent server (FastAPI + asyncio)
  │     ├── Agent loop (Anthropic API, SSE streaming)
  │     ├── Tools (bash, read, edit, git, python)
  │     └── SQLite session store
  └── Repos (local filesystem, full git access)
```

The server is the only backend. The browser frontend is served from the same process.
No iOS-specific execution constraints apply — the server has fork(), subprocess,
native git, any Python package.

## Data Model

### Chains

A **chain** is the fundamental user-visible unit — what the user resumes. Every new
conversation starts a new chain. Continuation sessions stay within the same chain.
Subtasks spawn child chains, forming a tree.

```sql
CREATE TABLE chains (
    id          TEXT PRIMARY KEY,
    parent_id   TEXT REFERENCES chains(id),  -- null for root chains
    repo_path   TEXT,
    title       TEXT,                         -- auto-generated from first message
    goal        TEXT,                         -- optional, set for task-directed chains
    status      TEXT DEFAULT 'active',        -- active | completed | aborted | blocked
    created_at  INTEGER NOT NULL,
    updated_at  INTEGER NOT NULL
);
```

### Sessions

A **session** is one continuous LLM context window within a chain. A chain contains
one or more sessions connected by continuation briefs.

```sql
CREATE TABLE sessions (
    id              TEXT PRIMARY KEY,
    chain_id        TEXT NOT NULL REFERENCES chains(id),
    sequence        INTEGER NOT NULL,   -- position within chain
    status          TEXT DEFAULT 'active',
    token_count     INTEGER DEFAULT 0,
    started_at      INTEGER NOT NULL,
    ended_at        INTEGER,
    summary         TEXT,               -- generated at session end
    UNIQUE(chain_id, sequence)
);
```

### Messages

```sql
CREATE TABLE messages (
    id          INTEGER PRIMARY KEY,
    session_id  TEXT NOT NULL REFERENCES sessions(id),
    sequence    INTEGER NOT NULL,
    role        TEXT NOT NULL,      -- user | assistant
    content     TEXT NOT NULL,
    created_at  INTEGER NOT NULL
);

CREATE VIRTUAL TABLE messages_fts USING fts5(
    content,
    content='messages',
    content_rowid='id'
);
```

### Tool Calls

```sql
CREATE TABLE tool_calls (
    id          INTEGER PRIMARY KEY,
    message_id  INTEGER NOT NULL REFERENCES messages(id),
    session_id  TEXT NOT NULL,
    tool        TEXT NOT NULL,
    input       TEXT NOT NULL,      -- JSON
    result      TEXT,               -- JSON, null while running
    exit_code   INTEGER,            -- for bash tool
    started_at  INTEGER NOT NULL,
    ended_at    INTEGER
);
```

### File Events

```sql
CREATE TABLE file_events (
    id          INTEGER PRIMARY KEY,
    session_id  TEXT NOT NULL,
    chain_id    TEXT NOT NULL,
    path        TEXT NOT NULL,
    event       TEXT NOT NULL,      -- read | write | edit
    diff        TEXT,               -- unified diff for edits
    created_at  INTEGER NOT NULL
);
```

### Continuation Briefs

```sql
CREATE TABLE continuation_briefs (
    id              INTEGER PRIMARY KEY,
    from_session_id TEXT NOT NULL REFERENCES sessions(id),
    to_session_id   TEXT REFERENCES sessions(id),   -- null until next session starts
    goal            TEXT,
    accomplished    TEXT NOT NULL,  -- JSON array of strings
    remaining       TEXT,           -- JSON array of strings
    key_context     TEXT,           -- JSON: [{type, path, session_id, message_id}]
    termination     TEXT NOT NULL,  -- JSON: {status, confidence, reason}
    created_at      INTEGER NOT NULL
);
```

**Termination status values:**

```json
{
  "status": "done | continuing | blocked | uncertain",
  "confidence": "high | medium | low",
  "reason": "all tests pass, refactor complete"
}
```

Hard cap reached → status is implicitly `continuing`, brief is generated by the server
from available context without agent input.

## Session Lifecycle

```
Chain created (first user message)
  │
  └── Session 1 starts
        │
        ├── tokens accumulating...
        │
        ├── Soft cap (~65% of context window)
        │     ├── user present → hint surfaced to user and agent
        │     └── user absent → hint to agent only; agent should wrap up
        │
        ├── Hard cap (~85% of context window)
        │     └── server forces session end, generates continuation brief
        │           from remaining context budget
        │
        └── Session end (any cause)
              ├── agent writes termination declaration
              ├── summarization pass (one LLM call)
              │     ├── session.summary (one line, for search)
              │     └── continuation_brief (if status != done)
              └── Session 2 starts (if continuing)
```

The gap between soft and hard cap (~20%) is the wrap-up budget: finish current tool
call + write termination declaration + generate continuation brief.

User presence is determined by time since last user message (configurable threshold,
default 5 minutes).

---

## Context Management — Ralph Loop Integration

A chain with a `goal` set supports autonomous iteration (ralph loop pattern):

- Agent runs, hits soft cap, writes termination declaration
- If `status: continuing` → server starts next session with continuation brief injected
- If `status: done` → chain marked completed
- If `status: blocked` → chain marked blocked, user notified, no continuation
- Hard cap always continues

**Max continuation depth** is configurable per chain (default: 10). Exceeding it
forces `blocked` status and requires user intervention.

Child chains can be spawned for subtasks. The agent emits a structured subtask
request; the server creates a child chain and starts a session within it.

## API

### Chains

```
GET    /chains                          list resumable chains (tree structure)
POST   /chains                          start new chain
GET    /chains/{id}                     chain detail + session list
PATCH  /chains/{id}                     update goal, status
GET    /chains/{id}/stream              SSE stream for active session
POST   /chains/{id}/message             send user message
POST   /chains/{id}/stop               interrupt active session
```

### Sessions

```
GET    /sessions/{id}                   session detail
GET    /sessions/{id}/messages          paginated messages
GET    /sessions/{id}/messages/{msg_id} single message (full tool call content)
```

### Search

```
GET    /search/chains?q=&repo=&status=&from=&to=
GET    /search/messages?q=&chain_id=&session_id=&from=&to=
GET    /files/{path}/chains             all chains that touched this file
```

### Filesystem (read-only from browser)

```
GET    /fs/tree?path=                   directory listing
GET    /fs/file?path=&from=&to=        file contents with optional line range
```

### Git

```
GET    /git/status?repo=               changed files, branch, ahead/behind
POST   /git/pull?repo=
POST   /git/push?repo=
POST   /git/clone                      {url, path}
```

## SSE Event Stream

The browser subscribes to `/chains/{id}/stream`. Events:

```json
{ "type": "text",         "content": "I'll add error handling..." }
{ "type": "tool_start",   "tool": "bash", "input": {"cmd": "cargo test"} }
{ "type": "tool_result",  "tool": "bash", "exit": 0, "output": "..." }
{ "type": "file_changed", "path": "src/parser.zig" }
{ "type": "cap_warning",  "level": "soft", "pct": 65 }
{ "type": "session_end",  "status": "continuing" }
{ "type": "session_start","session_id": "..." }
{ "type": "done",         "termination": {...} }
```

`file_changed` causes the browser to re-fetch the file via `/fs/file`.

## Tools

### read

Returns file content with line numbers prefixed. The line numbers become the
coordinate system for the edit tool.

```
Input:  { path: str, from_line?: int, to_line?: int }
Output: line-numbered text

  1  fn parseExpr(
  2    allocator: Allocator,
  3    tokens: []Token,
  4  ) !Expr {
```

### edit

Accepts unified diff format. Because the LLM read line numbers directly from the
read tool output, writing `@@` hunk headers requires no counting — line numbers are
looked up from what was already shown.

```
Input:  { path: str, diff: str }

@@ -3,1 +3,1 @@
-  tokens: []Token,
+  tokens: []const Token,
```

Context lines in the diff are used to validate the file hasn't changed since the
read. If validation fails, the tool returns an error and the agent must re-read.

### bash

```
Input:  { cmd: str, cwd?: str, timeout?: int }
Output: { stdout: str, stderr: str, exit: int }
```

Full shell access. Runs via asyncio subprocess. No sandboxing — this is a personal
tool on a trusted machine.

### python

```
Input:  { code: str, cwd?: str }
Output: { stdout: str, stderr: str, exit: int }
```

Runs in a subprocess with the system Python. Prefer bash for simple invocations.

### git_status / git_pull / git_push

Thin wrappers over subprocess git. Results are structured JSON rather than raw
terminal output.

### search_sessions

```
Input: {
  query?: str,        # FTS over message content
  repo?: str,
  file?: str,         # filter by file touched
  date_from?: str,
  date_to?: str,
  limit?: int
}
Output: [ChainSummary]  # small, context-friendly
```

### fetch_session

```
Input: {
  chain_id: str,
  session_id?: str,
  query?: str,          # FTS within session, returns matching messages + neighbours
  from_message?: int,
  to_message?: int,
  collapse_tool_calls?: bool  # default true
}
Output: SessionDetail
```

## Frontend

Browser PWA served from FastAPI. Three-pane horizontal scroll-snap layout:

```
[ Dir Browser ] [ File Viewer ] [ Chat ]
```

**Input bar** — pinned to bottom, present on all panes:

```
[ context chips (file:line-range, dir) ]  [ 🎙 mic ]  [ ↑ send ]
```

- Mic button triggers Web Speech API — no keyboard required
- Context chips auto-populated from the current pane; tap × to remove
- When agent is active: chips replaced with status + stop button

**File viewer** — read-only, syntax highlighted. When agent edits a file,
`file_changed` SSE event triggers a re-fetch and a brief green flash on changed lines.

**Dir browser** — file tree for the current repo. Tap file → jump to file viewer.
Files touched in the current session get a dot badge.

**Chat pane** — conversation history. Tool calls rendered as collapsed cards;
tap to expand. Tap a file edit card → jump to file viewer with that file open.

**Chain list** — sheet or sidebar showing resumable chains as a tree. Active chains
show session count and last activity. Completed chains are searchable.

## Open Questions

1. Auth — Tailscale handles network security but the server itself has no auth.
   Acceptable for personal use; add a static token header if sharing the machine.

2. Multiple repos — server should support switching between repos per chain.
   `repo_path` on the chain record handles this; the filesystem and git tools
   take it as a parameter.

3. Subtask spawning protocol — exact format for the agent to request a child chain.
   Likely a structured tool call: `spawn_subtask({ goal, context })`.

4. Session search tool availability — should the agent always have access to session
   search, or only when explicitly given it? Probably always available but with
   collapsed/summarised results by default.

5. Web Speech API quality on iOS Safari — worth prototyping early before committing
   to voice-first input.
